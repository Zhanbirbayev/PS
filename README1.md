# Terraform & Ansible Web Cluster — Cloud Engineer Test Task

Данный проект реализует **автоматизированное развертывание отказоустойчивой веб-инфраструктуры в облаке** с использованием подхода **Infrastructure as Code (IaC)**.

В основе решения лежит следующий стек:

- **Terraform** — создание инфраструктуры
- **Ansible** — настройка серверов
- **HAProxy** — балансировка трафика
- **Apache** — веб-серверы
- **MinIO / S3** — хранение Ansible-ролей

---

##  Архитектура решения

### Сетевая схема

Инфраструктура разворачивается во внутренней **«серой» сети**, а доступ в интернет обеспечивается через **Floating IP (FIP)**.

### Внешний доступ

Входящий HTTP-трафик поступает на публичный IP-адрес, привязанный к балансировщику:

- **Web Access:** http://185.102.75.187
- **HAProxy Stats:** http://185.102.75.187:8080/stats

**Данные для входа в статистику HAProxy:**

- **Username:** `admin`
- **Password:** `admin`

### Внутренний сегмент

Все узлы инфраструктуры взаимодействуют между собой через **изолированную внутреннюю сеть**, что повышает уровень безопасности и исключает прямой внешний доступ к backend-серверам.

### Источник сетевой конфигурации

Логика построения сети и подключения Floating IP основана на официальной документации **PS Cloud**:

🔗 https://static-docs.archive.pscloud.io/openstack_centos7_gray_network_fip.txt

## 🖥️ Состав инфраструктуры

Проект разворачивает следующие узлы:

- **1 Control Server** — управляющий сервер
- **1 HAProxy Server** — балансировщик нагрузки
- **3 Web Servers** — Apache веб-серверы

---

## ⚙️ Принцип работы

### Control Server

В данной архитектуре **Control Server** (IP: `213.155.21.223`) выполняет роль:

- **Bastion Host**
- **Management Node**
- **Execution Node для Ansible**

Именно этот сервер является **единой точкой управления всей инфраструктурой**.

### Основные функции Control Server

- запуск Ansible-плейбуков;
- подключение к внутренним узлам по SSH;
- загрузка ролей из S3 / MinIO;
- распаковка ролей в `/etc/ansible/`;
- применение конфигурации к узлам:
  - `web1`
  - `web2`
  - `web3`
  - `haproxy`

---

## ☁️ Хранение ролей и интеграция с S3

Для хранения Ansible-ролей используется **S3-совместимое хранилище MinIO**.

Control Server использует утилиту **mc** для:

- авторизации в MinIO;
- скачивания архивов с ролями;
- подготовки окружения для деплоя.

### Используемые переменные

В плейбуке используются следующие параметры:

- `s3_host` — адрес S3 / MinIO хранилища
- `s3_username` — логин для доступа
- `s3_password` — пароль для доступа

## 🚀 Развертывание инфраструктуры

### 1. Инициализация Terraform

Скачивание необходимых провайдеров для работы с PS Cloud:

```bash
terraform init
```

### 2. Планирование

Проверка ресурсов, которые будут созданы:

- виртуальные машины;
- сети;
- правила доступа;
- Floating IP.

```bash
terraform plan
```

### 3. Применение

Создание инфраструктуры в облаке:

```bash
terraform apply
```

## 🛠️ Конфигурация серверов через Ansible

После того как Terraform завершает создание виртуальных машин, **Ansible** берет на себя настройку серверов и разворачивание сервисов.

---

## ▶️ Работа с Ansible на Control Server

Для повторного запуска конфигурации необходимо подключиться к **Control Server** и выполнить следующие команды.

### Переход в рабочую директорию

```bash
cd ~/ansible-project
```

### Проверка доступности узлов

Проверяем SSH-доступ до всех серверов:

```bash
ansible all -m ping -i hosts.ini
```

### Запуск полной конфигурации

Основной сценарий:

- скачивает роли из S3;
- распаковывает их;
- применяет конфигурацию ко всем узлам;
- разворачивает рабочий кластер.

```bash
ansible-playbook -i hosts.ini site.yml
```

## ✅ Результат развертывания

После успешного выполнения проекта вы получаете:

### Apache Web Cluster

Три веб-сервера Apache с персонализированными страницами:

```text
Welcome to <hostname>
```

При каждом обновлении страницы запрос может обрабатываться разным сервером.

### HAProxy Load Balancer

Балансировщик **HAProxy** распределяет входящий трафик между backend-серверами.

Это обеспечивает:

- равномерную нагрузку;
- отказоустойчивость;
- централизованную точку входа в приложение.

## 📂 Пример логики развертывания

1. Terraform создает сеть, виртуальные машины и Floating IP  
2. Control Server получает доступ к инфраструктуре  
3. Control Server скачивает Ansible-роли из MinIO / S3  
4. Ansible настраивает HAProxy и Apache-серверы  
5. Пользователь получает доступ к веб-кластеру через публичный IP
