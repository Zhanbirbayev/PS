# Infrastructure as Code: Web Cluster with Load Balancer

Решение демонстрирует автоматизированное развертывание **веб-кластера из 5 виртуальных машин** с использованием подхода **Infrastructure as Code (IaC)**.

В основе решения лежит следующий стек:

- **Terraform** — создание инфраструктуры и автоматический запуск конфигурации
- **Ansible** — настройка серверов
- **HAProxy** — балансировка нагрузки
- **Apache** — веб-серверы
- **MinIO / S3** — хранение Ansible roles

---

##  Описание задания

### Terraform
Необходимо:

- создать **5 виртуальных машин на CentOS**
  - **1 Control VM**
  - **1 HAProxy VM**
  - **3 Apache VM**
- назначить **Floating IP** на HAProxy
- задать **фиксированные локальные (серые) IP-адреса**
- установить необходимые пакеты на **Control VM**
- автоматически запустить **Ansible-конфигурацию**

### Ansible
Необходимо:

- настроить:
  - **1 HAProxy**
  - **3 Apache-сервера**
- создать страницу:
  - `/opt/html/index.html`
  - содержимое: `Welcome to <hostname>`
- **не отключать SELinux**
- хранить **Ansible roles в S3**

##  Общая архитектура решения

### Сетевая схема

Инфраструктура разворачивается во внутренней **«серой» сети**, а доступ в интернет обеспечивается через **Floating IP (FIP)**.

### Внешний доступ

Входящий HTTP-трафик поступает на публичный IP-адрес, привязанный к **HAProxy**:

- **Веб-доступ:** http://185.102.75.187
- **Статистика HAProxy:** http://185.102.75.187:8080/stats

**Данные для входа в статистику HAProxy:**

- **Username:** `admin`
- **Password:** `admin`

### Внутреннее взаимодействие

Все узлы работают во внутреннем изолированном сегменте:

- `haproxy`
- `web1`
- `web2`
- `web3`
- `control`

Такой подход повышает безопасность и исключает прямой внешний доступ к backend-серверам.

### Основа сетевой логики

Сетевая схема реализована на основе документации **PS Cloud**:

https://static-docs.archive.pscloud.io/openstack_centos7_gray_network_fip.txt

##  Состав инфраструктуры

Проект разворачивает следующие виртуальные машины:

- **1 Control Server** — управляющий сервер
- **1 HAProxy Server** — балансировщик нагрузки

### Фиксированные внутренние IP-адреса

| Хост      | Роль                  | Внутренний IP  |
|-----------|-----------------------|----------------|
| control   | Управляющий сервер    | `192.168.0.50` |
| haproxy   | Балансировщик         | `192.168.0.10` |
| web1      | Apache backend        | `192.168.0.11` |
| web2      | Apache backend        | `192.168.0.12` |
| web3      | Apache backend        | `192.168.0.13` |

##  Как работает решение

### Что делает Terraform

Terraform отвечает за:

- создание сетевой инфраструктуры;
- создание всех 5 виртуальных машин;
- назначение фиксированных внутренних IP-адресов;
- выделение и привязку Floating IP;
- подготовку **Control Server**;
- копирование Ansible-файлов на Control VM;
- **автоматический запуск Ansible после поднятия инфраструктуры**.

### Что делает Ansible

Ansible отвечает за:

- настройку **HAProxy**
- настройку **3 Apache-серверов**
- создание тестовой страницы:

```text
Welcome to <hostname>
```

- применение конфигурации ко всем backend-узлам
- работу **без отключения SELinux**

---

##  Интеграция с S3 / MinIO

Для хранения Ansible-ролей используется **S3-совместимое хранилище MinIO**.

Control Server может использовать утилиту **mc** для:

- авторизации в MinIO;
- скачивания архивов с ролями;
- подготовки конфигурации перед запуском деплоя.

### Используемые переменные для доступа к S3

В плейбуке используются следующие параметры:

- `s3_host` — адрес S3 / MinIO хранилища
- `s3_username` — логин для доступа
- `s3_password` — пароль для доступа

> В текущей реализации базовые Ansible-файлы (`hosts.ini`, `site.yml`) передаются на **Control Server** автоматически через Terraform.  
> Дополнительные роли могут загружаться из **S3 / MinIO** уже во время выполнения playbook.

##  Развертывание

### 1. Инициализация Terraform

Скачивание необходимых провайдеров:

```bash
terraform init
```

### 2. Просмотр плана

Проверка ресурсов, которые будут созданы:

```bash
terraform plan
```

### 3. Применение конфигурации

Создание инфраструктуры и автоматический запуск Ansible:

```bash
terraform apply
```

---

##  Автоматический запуск Ansible

После того как Terraform завершает создание всех виртуальных машин, он **автоматически подключается к Control Server и запускает Ansible**.

### Terraform выполняет следующие действия:

1. Подключается по SSH к **Control VM**
2. Передает на сервер:
   - `hosts.ini`
   - `site.yml`
   - SSH-ключ для доступа к внутренним узлам
3. Устанавливает необходимые пакеты:
   - `ansible`
   - `git`
   - `curl`
   - `tar`
4. Запускает playbook:

```bash
ansible-playbook -i hosts.ini site.yml
```

Таким образом, **после `terraform apply` дополнительная ручная настройка не требуется**.

##  Повторный ручной запуск (опционально)

При необходимости конфигурацию можно повторно запустить вручную на **Control Server**.

### Подключение к Control Server

```bash
ssh -i ~/.ssh/pscloud_key centos@<CONTROL_PUBLIC_IP>
```

### Переход в директорию Ansible

```bash
cd /home/centos/ansible
```

### Проверка доступности узлов

```bash
ansible all -m ping -i hosts.ini
```

##  Заключение

Данный проект демонстрирует практическую реализацию подхода **Infrastructure as Code** для построения **отказоустойчивого веб-кластера** в **PS Cloud**.

Решение обеспечивает:

- **автоматическое развертывание**
- **автоматическую конфигурацию**
- **сетевую изоляцию**
- **балансировку нагрузки**
- **повторяемость деплоя**

Также проект показывает, как можно объединить **Terraform и Ansible в единый сценарий развертывания**, где создание инфраструктуры и настройка серверов выполняются в рамках одного процесса.
